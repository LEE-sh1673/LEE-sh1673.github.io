Boxing/UnBoxing

Boxing과 UnBoxing은 C#에서 중요한 개념입니다. C#의 형식 시스템은 다음과 같은 세가지 형식을 포함합니다.

  • 값 형식(Value Type) – E.g. int, char, etc.
  • 참조 형식(Reference Type) – Object
  • 포인터 형식(Pointer Type)

기본적으로 값 형식은 참조 형식으로 바꿀 수 있고, 그 반대도 마찬가지 입니다. Boxing과 UnBoxing은 어떠한 형식이든 Object로써 취급될 수 있게 하는 형식 시스템의 통합적인 관점 가능하게 합니다.

값 형식과 참조 형식의 차이점

본격적으로 Boxing과 UnBoxing에 대해 알아보기 전에, 알아야 할 것들이 있습니다. 값 형식은 프로그램에 어디에 저장되고, 참조 형식은 또 어디에 저장될까요? 이는 Boxing과 UnBoxing에 필요한 선수 지식이기 때문에 먼저 이에 대해 설명하고자 합니다. 다음은 StackOverflow에서 가져온 질문입니다.

then I can create an bicycle object:
Then this bicycle object should be stored in heap. But I don't understand how heap exactly store these instance variables and methods such as speed and gear.

I understand that heap should be implemented as tree. So how the object is stored in a tree? Also when you use bicycle.speed to find the value of speed, what will be time complexity?

StackOverflow - How is an object stored in heap?

즉, 이 사람이 질문한 것은 참조 형식의 변수 object가 정확히 어떻게 heap 영역에 저장되는 것인지를 물어본 것입니다. (그 외에 메서드는 또 어떻게 저장되는지와 heap의 구조와 연관된 질문, 시간 복잡도 등도 물어봤지만 이건 넘어가도록 합시다. 자세한 내용이 궁금하시다면 위의 링크를 타고 가서 보시면 됩니다.)

거기에 달린 답변은 아래와 같습니다.

... the local variables and parameters go on the stack, and whenever you new something it goes into the heap. It's called a stack because values are pushed onto it when you declare it or call a function, and popped off and they go out of scope.
...
So in your example:
  • bicycle is a reference to the heap memory address, today in most languages/systems it will cost you either 32 and 64 bits on the stack.
  • The new allocates memory in the heap. The compiler figures out the size of Bicycle and creates assembly/machine code that allocates the amount of memory it requires.
This is how the memory looks after this line: 
                    +--------------+
|              |    | Bicycle obj  |
|              |    |--------------|
|              |    |              |
|              |    |              |
|--------------|    |              |
| bicycle ref  |    |              |
+--------------+    
     Stack                Heap
More specifically, since the Bicycle class has two instance variables (or fields as they are called in Java) and both are ints, and an int in Java is 32 bits or 4 bytes, the size of your Bicycle object is 4 bytes * 2 fields = 8 bytes. 
                   +-------------+
|             |   0| gear        | 
|             |   4| speed       |
|             |    |-------------|
|             |   8|             |
|-------------|  12|             |
| bicycle=0x4 |    |             |
+-------------+    
     Stack               Heap

답변에서는 지역변수와 매개변수는 스택에 저장되고 new연산자를 쓸때 인스턴스는 힙에, 그 인스턴스를 참조하는 bicycle은 스택에 저장된다는 것입니다.

또한 힙에 저장되는 인스턴스의 크기는 클래스의 맴버 변수 2개가 모두 int형이고 대부분의 언어 체계에서 4바이트의 크기이기 때문에 결국 8바이트의 크기를 힙 영역에 할당 받는다는 이야기입니다.

정리하자면 아래와 같습니다.

  • 일반 값 형식의 변수나 메서드의 매개변수들은 스택에 저장됩니다.
  • 참조 형식의 변수들은 new 연산자 호출 시 인스턴스는 힙 영역에 그에 대한 참조는 스택영역에 저장됩니다.

왜 인스턴스에서는 static 메서드에 접근할 수 없나요?

클래스 안에 정의된 static 키워드가 붙은 메서드는 인스턴스에서 접근할 수 없습니다. 이유가 뭘까요?

아래 코드를 보면,

srcStr이 참조하고 있는 문자열에 World! 문자열을 붙여서 dstStr에 저장한 후 출력하는 코드입니다. 위의 코드는 문제없이 작동합니다. 위의 코드를 아래와 같이 바꾸고 출력해 보면 어떨까요?

실행시키면 아래와 같은 에러 메세지가 출력됩니다.

error CS0176: Static member `string.Concat(string, string)‘ cannot be accessed with an instance reference, qualify it with a type name instead

즉, 인스턴스 참조로는 static 메서드를 접근할 수 없으니 형식 이름을 사용하라는 의미입니다. 이유가 뭘까요?

static 선언시 선언과 동시에 정적 영역의 메모리에 저장됩니다. 만약 string 타입의 변수 srcStr가 이러한 메서드들을 사용한다면 이미 정의된 메서드이기 때문에 변수이름.메서드의 형식으로 사용할 수 없습니다.

즉 string 타입의 변수 str를 선언할 때, 내부적으로 정의된 메서드들을 새롭게 static 영역에 저장하려고 시도하지만 이미 정의된 메서드이기 때문에 불가능합니다.

따라서 이러한 메서드들을 사용할 때는 형식이름.메서드의 형식으로 호출해야 합니다.

예를 들어봅시다. 클래스를 집으로 가정하면, 그 안에서 메서드는 집에서 쓸 수 있는 기능으로 비유할 수 있습니다.

여기서 static이 붙은 메서드와 그렇지 않는 메서드의 차이는 static이 붙은 메서드는 모든 집에서 공용으로 사용하는 기능이라고 할 수 있습니다. (예: 주민들을 대상으로 재공하는 주민센터의 헬스장)

그렇지 않은 메서드들은 그 집집마다 달려있는 기능, 예를 들어 보일러를 튼다든가 전기가 들어온다거나… 하는 기능들이라고 할 수 있습니다.
